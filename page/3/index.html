发表于 2019-11-14

逻辑语言

一种描述A，要么为真，要么为假。那么根据描述A，我们可以将世界分为两种，描述A为真的世界，和描述A为假的世界。
一种描述B，要么为真，要么为假。那么根据描述B，我们可以将世界分为两种，描述B为真的世界，和描述B为假的世界。

我们所需要的结果，就是众多世界中的一种或者一部分。这个结果，是将描述A和描述B的真假组合在一起之后得到一个世界，或者一部分世界。

逻辑语言就是用来表达描述A为真假这种信息的。换句话说，我们用逻辑语言来准确表达我们已知的部分信息，或者全部信息，不多也不少。

逻辑蕴含

如果一些逻辑语言在全部的世界中都是对的，那么从这些逻辑语言得出的结论，在全部的世界中也是正确的。

所以我们如果能证明这些逻辑语言，在全部的世界中都是对的，那么我们就可以从这些逻辑语言得出正确的结论。通过这些结论，我们最后可以确定一种世界
或者一部分世界。

逻辑推理

在通常的情况下，全部的世界到底有多少，我们不清楚，有很多或者数目就是无限的。我们只有少数的世界。所以我们没法证明这些逻辑语言，在全部的世界中都是对的。

我们需要用到一些逻辑推理方法，运用一些思考步骤，证明逻辑语言的正确性，并得到正确的结论。

逻辑推理方法有：
归纳推理，从特殊推导到一般。
溯因推理，是从结论推导原因。
类比推理，是从相同的前提来推导结论。

以下摘自维基百科：

https://zh.wikipedia.org/wiki/%E9%80%BB%E8%BE%91%E6%8E%A8%E7%90%86

邏輯推理中有三種方式：演绎、归纳和溯因。給定前提、結論和規則，而前提導致結論，例如：

前提：下雨了

规则：下雨使草地变湿

结论：草地是湿的

演繹用來決定結論。它使用規則和前提來推導出結論。數學家通常使用這種推理。舉例：”若下雨，則草地會變溼。因為今天下雨了，所以今天草地是溼的。”。

歸納用來決定規則。它藉由大量的前提和結論所組成的例子來學習規則。科學家通常使用這種推理。舉例：”每次下雨，草地都是溼的。因此若明天下雨，明天草地就會變溼。”。

溯因用來決定前提。它藉由結論和規則來支援前提以解釋結論。診斷和偵探通常使用這種推理。舉例：”若下雨，草地會變溼。之所以草地是溼的，因为曾下過雨。”。

逻辑表达

可以使用逻辑语法，来表达逻辑的前提，规则和结论的推导过程。

p1 ∧ … ∧ pk ⇒ q1 ∨ … ∨ ql

r1 ∧ … ∧ rm ⇒ s1 ∨ … ∨ sn

p1 ∧ … ∧ pk ∧ r1 ∧ … ∧ rm ⇒ q1 ∨ … ∨ ql ∨ s1 ∨ … ∨ sn

如果某个符号在等号一边的某个表达式中出现了，并且也在等号另一边的另一个表达式出现了，那么这个符号可以在计算时被删除。
如果某个符号，在等号一边的某个表达式中，出现了多次，那么这个符号的多次出现可以被一次出现代替。

例如：
p ⇒ q

m ∧ r ⇒ p ∨ q

由于p在等号左边的表达式，和右边的另一个表达式中都有得到，所以我们可以得到：

m ∧ r ⇒ q ∨ q

而从上面，我们可以删除q符号，因为它在等号右边的表达式中出现了多次。

m ∧ r ⇒ q ∨ q

m ∧ r ⇒ q

自动化推理

由于我们使用了逻辑语法来表达逻辑的前提，规则，那么推导的过程可以利用机器自动进行了。

Privilege level and user access level

privilege level可以访问所有资源；

user access level不能访问特殊寄存器，debug寄存器，系统控制寄存器，以及MPU设置的禁止内存区域，其他区域都可以访问。

当处理器进入异常处理程序时，就处于handler模式下，是previlege level；

当处理器执行用户态程序时，就处于thread模式下，此时可以处于previlege level或者user access level。

EAP的含义，扩展认证协议。是个可扩展的协议框架。
EAP是个应用层的协议，EAP之上可以承载其他的认证计费协议。
EAP在二层LAN上，例如ethernet侧，就是封装在802.1X报文的payload部分。
EAP在三层IP上，就是封装在radius协议报文里面，而RADIUS则是封装在UDP报文里面。（RADIUS终结）

CHAP是使用MD5对用户密码进行了摘要计算。

• 问题
 CHAP使用MD5计算摘要，因此Radius服务器是无法还原出用户输入的原始口令的，基于此，CHAP协议只支持使用本地数据源进行认证。 为什么呢？

2、服务器会根据配置的认证协议集合和默认协议，选择一个认证协议，并发送该认证协议的radius challenge报文给客户端；

        3.2.2 不支持，就回复failure。

03_network/2012-03-10-802.1X和Radius协议

发表于 2019-11-14

EAP协议

EAP（Extensible Authentication Protocol）为可扩展身份验证协议，是一系列验证方式的集合。它是一种框架协议，在其基础上，实现了多种认证、计费和授权协议，例如GTC、MD5和MSCHAPV2等。

在远程接入认证的应用场景中，例如家庭宽带拨号，企业园区网络设备接入和认证，
用户通过终端发起认证，由认证服务器对用户身份进行校验和验证，并授权用户访问网络服务。

EAP协议，就是为了服务这种远程接入认证的应用而设计的。在这种应用场景中，涉及到的设备可以分为3类：终端，网络转发设备，认证服务器(Radius服务器）。

终端上安装了客户端软件，一般指电脑和手机，还有各种联网打印设备。

网络转发设备一般指交换机，或者AC。也叫Radius客户端。

认证服务器一般指Raidus服务器。

终端和网络转发设备之间一般是局域网，例如以太网或者无线局域网，EAP协议报文在局域网上由802.1X协议报文封装，也叫EAPOL（EAP Over Lan）。
网络转发设备和认证服务器之间，EAP协议报文在Radius协议报文中封装。这里分为两种情况，EAP中继和EAP终结。

对于EAP中继，是指一方，网络转发设备将终端发送过来的802.1X EAP报文，原封不动的封装在Radius报文的EAP Message属性中，再发送给Radius服务器。另一方，Radius服务器发送的带有EAP Message属性的Radius报文，交换机将EAP报文内容提取出来，封装在802.1X协议的报文中，发送给终端。

对于EAP终结，是指标准Radius协议本身定义了PAP和CHAP属性。终端发送过来的信息，网络转发设备直接将这些信息以Radius属性的形式发送给Radius服务器，而不需要再封装到EAP属性中。EAP终结适用的协议，一般没有定义客户端和网络转发设备之间的交互流程，只是定义了网络转发设备和Radisu服务器之间的交互流程。

安装文件的目录介绍

bin目录，存放启动和命令脚本文件。start.bat文件。

02_java/2016_04_09-JSP隐式对象

隐式对象

在JSP文件中，可以无需显式的声明和创建一些对象，就可以直接使用它们。这些对象，叫做隐式对象。

JSP所支持的九大隐式对象：

对象 描述
request HttpServletRequest类的实例
response HttpServletResponse类的实例
out JspWriter类的实例，用于把结果输出至网页上
session HttpSession类的实例
application ServletContext类的实例，与应用上下文有关
config ServletConfig类的实例
pageContext PageContext类的实例，提供对JSP页面所有对象以及命名空间的访问
page 类似于Java类中的this关键字
Exception Exception类的对象，代表发生错误的JSP页面中对应的异常对象

request和response对象

每当1个客户端请求JSP页面时，就会产生1个对应的request和response对象。

request对象代表请求，该对象提供了一系列的方法用来获取HTTP头信息，cookies，HTTP方法等。

response对象代表响应，该对象提供了一系列的方法用来设置响应的内容，例如cookies，时间戳，HTTP状态吗等。

out对象

可以使用out对象直接往response对象里面写入内容。这些内容直接输出到响应网页中。

session对象

session对象和Java Servlets中的session对象有一样的行为。

session对象用来跟踪在各个客户端请求间的会话。

application对象

这个对象在JSP页面的整个生命周期中都代表着这个JSP页面。这个对象在JSP页面初始化时被创建，随着jspDestroy()方法的调用而被移除。

通过向application中添加属性，则所有组成您web应用的JSP文件都能访问到这些属性。

page 对象

这个对象就是页面实例的引用。它可以被看做是整个JSP页面的代表。

page 对象就是this对象的同义词

过滤器

JSP过滤器，用于：

  • 在客户端的请求访问后端资源之前，拦截这些请求。
  • 在服务器的响应发送回客户端之前，处理这些响应。

JSP过滤器，有多种类型：

  • 身份验证过滤器（Authentication Filters）。
  • 数据压缩过滤器（Data compression Filters）。
  • 加密过滤器（Encryption Filters）。
  • 触发资源访问事件过滤器。
  • 图像转换过滤器（Image Conversion Filters）。
  • 日志记录和审核过滤器（Logging and Auditing Filters）。
  • MIME-TYPE 链过滤器（MIME-TYPE Chain Filters）。
  • 标记化过滤器（Tokenizing Filters）。
  • XSL/T 过滤器（XSL/T Filters），转换 XML 内容。

HTTP响应头

1个服务器回复的HTTP响应报文，包含以下头部信息：

响应头 描述
Allow 指定服务器支持的request方法（GET，POST等等）
Cache-Control 指定响应文档能够被安全缓存的情况。通常取值为 public，private 或no-cache 等等。 Public意味着文档可缓存，Private意味着文档只为单用户服务并且只能使用私有缓存。No-cache 意味着文档不被缓存。
Connection 命令浏览器是否要使用持久的HTTP连接。close值 命令浏览器不使用持久HTTP连接，而keep-alive 意味着使用持久化连接。
Content-Disposition 让浏览器要求用户将响应以给定的名称存储在磁盘中
Content-Encoding 指定传输时页面的编码规则
Content-Language 表述文档所使用的语言，比如en， en-us,，ru等等
Content-Length 表明响应的字节数。只有在浏览器使用持久化 (keep-alive) HTTP 连接时才有用
Content-Type 表明文档使用的MIME类型
Expires 指明啥时候过期并从缓存中移除
Last-Modified 指明文档最后修改时间。客户端可以 缓存文档并且在后续的请求中提供一个 If-Modified-Since请求头
Location 在300秒内，包含所有的有一个状态码的响应地址，浏览器会自动重连然后检索新文档
Refresh 指明浏览器每隔多久请求更新一次页面。
Retry-After 与503 (Service Unavailable)一起使用来告诉用户多久后请求将会得到响应
Set-Cookie 指明当前页面对应的cookie

JSP Responce对象

JSP的Responce对象，提供了一系列的方法，用于操作HTTP响应头部。

可以添加新的cookie或时间戳，还有HTTP状态码等等。

例如：

  • void addCookie(Cookie cookie) 添加指定的cookie至响应中
  • void addIntHeader(String name, int value) 添加指定名称的响应头和int值
